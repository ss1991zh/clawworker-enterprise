# 授权吊销模型与已知局限

> 面向合规尽调 / 客户安全评审。回答一个尖锐问题:**"你们说授权可吊销,那客户端离线时,吊销到底多久生效?能不能被绕过?"**
> 诚实前提:本产品把**私钥 sk 分发到客户端本地**,解密是**纯本地运算**。任何把密钥放本地的方案,
> 都无法在密码学上阻止一个"已拿到密钥、且离线"的人解密自己手上的密文。因此"吊销"在本产品里是
> **策略 + 有界时间 + 审计**三层控制,而非"密码学实时阻断"。下文把边界讲清楚,不夸大。

---

## 一、吊销是怎么触发与生效的(闭环链路)

1. **管理端吊销**:admin 在主机后台对某用户执行吊销 → `AuthorizationManager.revoke(username)`
   把该用户的 `user_authorization` 标记 `revoked=True`(`host/cert_manager.py`)。
2. **主机侧即时拒绝**:被吊销用户
   - 重新登录 → `user_manager.login` 校验账户状态 + 授权 `is_valid()`,吊销即拒登录、并自动禁用账户;
   - 拉取授权 `/auth/user_authorization` → `is_valid()` 为假 → 404;
   - 携旧 token 调任何主机端点 → `get_current_session` 校验失败 → 401。
3. **客户端侧收敛离线窗口**:解密(把密文还原成明文)是敏感动作。客户端在**所有解密入口**
   (交互解密 `decrypt_decision`、事后解密 `decrypt_file`、定时任务批量解密 `scheduled_tasks/decrypt`)
   前置**会话新鲜度门** `_session_fresh()`:
   - 会话 `expires_at` 未过 → 放行(主机在会话 TTL 内核验过身份);
   - 会话已过 → **拦截解密**,提示"请重新登录后再解密"。重登时主机应用第 2 步的吊销判定。
   - **fail-closed**:离线且会话已过 → 无法重登 → 解密被拦。

**净效果**:被吊销用户能继续用**已有明文/密文**的时间上界 = **会话 TTL(默认 8 小时,`user_manager.SESSION_TTL`)**。
过了这个窗口,产品的正常解密路径一律要求回主机重登,吊销随即生效。

---

## 二、诚实的已知局限(不夸大)

1. **密码学层面无法阻止"离线 + 已持密钥"者解密**。sk 在客户端本地,一个技术用户绕过本产品 UI、
   直接调用 HE 库,仍能解密自己机器上的密文。**这不是本产品的 bug,是"密钥下发到端"架构的固有属性**——
   任何本地加解密产品(含全盘加密、本地密码库)都有同样性质。吊销的意义是:**撤销后不再下发新密钥/新密文、
   收敛正常路径、并留审计**,而非追回已在对方机器上的数据。
2. **吊销对"已经解出的明文 Excel"无效**。已导出到用户机器的明文文件不受吊销影响(与任何文件分发一致)。
3. **会话新鲜度门只约束本产品客户端**,不约束绕过客户端的直接调用(见第 1 条)。

---

## 三、缓解措施(把上界压到可接受)

| 杠杆 | 现状 / 建议 | 效果 |
|---|---|---|
| **会话 TTL** | 默认 8h(`SESSION_TTL`)。高敏客户可调短(如 1–2h) | 直接决定离线吊销延迟上界 |
| **授权有效期(SDK)** | user_authorization 自带有效期;签发时**取短**(如按月/按周) | 授权过期后 initSK 失败 → 密态全线停摆,是第二道 TTL |
| **密钥轮换** | 吊销后由 admin 轮换 sk/evk,旧密文用新密钥重签发 | 让对方手上的旧密文/旧密钥失去后续价值 |
| **审计留痕** | 每次解密经授权门并入不可篡改审计链(`docs`/可信审计) | 吊销后仍可追溯"谁在何时解过什么" |
| **在线强校验(可选增强)** | 未来可加"每次解密前 ping 主机核验 revoked",牺牲离线可用性换实时吊销 | 把上界压到 ~0,但要求解密时在线 |

---

## 四、给客户的一句话结论

> **吊销在正常使用路径上是有界实时的**:管理端一点吊销,被吊销用户在**≤ 会话 TTL(默认 8 小时)**内
> 就无法再通过客户端解密,且全程有审计。**我们不宣称能密码学阻止一个已持密钥且离线的人解密自己的密文**——
> 那是"密钥下发到端"架构的物理边界;对此的答案是短 TTL + 密钥轮换 + 审计,而非空话。
> 需要"实时吊销(解密前在线核验)"的高敏场景,可开启在线强校验模式(牺牲离线可用性)。
